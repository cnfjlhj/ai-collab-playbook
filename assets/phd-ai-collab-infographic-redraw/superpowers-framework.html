Superpowers：把 Coding Agent 变成工程化执行系统

它不是一组零散的技能目录，而是一套让 agent 先选流程、再对齐设计、再隔离执行、 最后经由 review、验证与分支收尾完成交付的 discipline system。

四条核心思想

不是 vibe coding 先流程后动作
A Skills First

using-superpowers 先判断该进哪条 lane， 不允许 agent 直接凭感觉开工。

B Design Before Implementation

brainstorming 和 writing-plans 先把模糊需求压成批准过的设计与可执行计划。

C Evidence Over Guessing

test-driven-development、 systematic-debugging、 verification-before-completion 都在禁止“凭感觉”写、修、报喜。

D Lifecycle, Not One-Off Output

using-git-worktrees、 review 技能和 finishing-a-development-branch 管的是完整交付生命周期，不只是代码片段。

主流程总线

从路由到收尾 每一步都知道自己在干什么
1

Route the Task

using-superpowers

先判断该进哪个 workflow，再回应、再行动。

2

Converge the Design

brainstorming

一次一个问题，比较 2-3 条路径，直到用户批准设计。

3

Turn Spec into Plan

writing-plans

把设计压成精确文件路径、测试步骤、命令和预期输出。

4

Prepare Clean Workspace

using-git-worktrees

在隔离分支和隔离目录里开工，并先确认 baseline 是干净的。

5

Execute with Guardrails

executing-plans subagent-driven-development

按计划执行；若平台支持 subagents，优先用 controller + reviewers 流水线。

6

Review and Interpret

requesting-code-review receiving-code-review

review 不是礼仪，而是质量闸门；反馈先核实，再实施。

7

Verify and Close

verification-before-completion finishing-a-development-branch

没有 fresh evidence 不许报喜；最后明确 merge / PR / keep / discard。

执行模式选择

线性执行 子代理流水线 真正独立时再并行

executing-plans

单会话线性执行模式。先批判性读 plan，再逐任务落实。

subagent-driven-development

每个 task 一个 implementer，先做 spec review，再做 code quality review。

dispatching-parallel-agents

只有在问题域真正独立、无共享状态、不会互相污染时才并行。

三个“别凭感觉”的纪律

写功能 修 bug 报完成

test-driven-development

先让测试红，再写最少代码变绿；没看见它先失败，就不知道它在不在测对的东西。

systematic-debugging

先找 root cause，再提 fix。不能一看到症状就开始补丁式乱修。

verification-before-completion

先找证明命令，真跑、真看、真判断，再说“通过了”“修好了”“完成了”。

Review 与系统演化

早 review 不盲从 方法论继续长技能

requesting-code-review

尽早 review，经常 review，让问题在扩散前被拦下来。

receiving-code-review

把 review 当技术输入，不把“你说得对”当作默认姿态。

writing-skills

把 recurring workflow 变成可发现、可触发、可验证的新 skill，让系统能复制自己。
